## 一、glibc 2.42+ 落点策略

> 背景：2.42（~2025-02）补 large_bin_attack；2.43 补 fastbin_reverse_into_tcache / fastbin_dup。**失去两大任意写原语后**，2.42+ 题的"堆地址写到任意地址"要换路。

### 1.1 tcache_metadata_hijacking（2.42+ 首选任意分配原语）

**结构**：`tcache_perthread_struct` 是堆上**第一个**分配的 chunk（通常 0x290 大小），位于 `heap_base`：
```
counts[64]   : uint16_t × 64 = 0x80   (每个 tcache bin 的计数)
entries[64]  : ptr     × 64 = 0x200   (每个 tcache bin 的头指针)
```
**关键认知**：`entries[]` 里存的是**原始指针**（未 safe-linking）。所以只要能写到 `tcache_perthread_struct`，**不需要堆泄漏绕过 safe-linking**——这是它比 tcache_poisoning 强的地方。

**具体步骤**（核心：把"任意写"升级为"任意地址分配"）：
1. 拿到一个能触及 `tcache_perthread_struct` 的写原语（典型途径见下）。
2. 改 `entries[tc_idx] = target_addr`，同时把对应 `counts[tc_idx]` 写成非 0（让 tcache 认为该 bin 非空）。
3. 下一次 `malloc(对应大小)` 直接返回 `target_addr`（`tcache_get` 返回的就是 `entries[tc_idx]` 原值；后续才 REVEAL next，不影响首次取回）。
4. 拿到任意地址分配后，往目标位置写伪造内容（IO 结构 / exit_funcs / `_dl_rtld_lock_recursive` 等），再触发 exit。

**如何触及 tcache metadata**（它是堆首，普通堆块在它之后，向前溢出打不到）：
- **House of Water 路线**：smallbin 改 fd/bk 低位（共享 ASLR 第二 nibble）指向 `tcache_perthread_struct`，排空 tcache 后从 smallbin 取回 → 分配到 metadata 上。【how2heap 已收录，glibc 2.36-2.39，原理在 2.42+ 仍成立】
- **tcache_poisoning 反打**：有堆泄漏时，毒化 fd 指向 `heap_base`（tcache 结构体），malloc 取回后直接覆盖 entries。
- **越界写回卷**：极少数题堆布局允许后块溢出回卷到 metadata（少见）。

### 1.2 2.42+ 落点决策树（large_bin_attack 没了之后）

| 手上原语 | 推荐落点 | 为什么 |
|---------|---------|--------|
| 单次任意写（任何来源） | **`_rtld_global._dl_rtld_lock_recursive`** | 无加密保护，写一次即可劫持，exit 触发。**2.42+ 最干净的落点** |
| 任意写 + 能泄漏 pointer guard | `__exit_funcs`（atexit 链） | fn 受 PTR_MANGLE 保护，需泄漏 TLS 中的 pointer guard 算值 |
| 任意写 + 想做 FSOP | 改 `_IO_list_all`/伪造 `_IO_FILE` + House of Apple 链 | 2.42 前靠 large_bin_attack 把堆地址写进 `_IO_list_all`；2.42 后**改用 tcache_metadata_hijack 直接在目标地址分配伪造块** |
| 仅 UAF / 无 free / 无写 | House of Tangerine（sysmalloc→_int_free 进 tcache）→ tcache poison → metadata_hijack | 不需要 free 函数，2.34+ 通用 |
| UAF + 堆泄漏 | House of Water → metadata_hijack → 任意分配 | smallbin 变体，无需爆破 |
| 堆块紧邻 TLS | 覆盖 TLS 的 pointer guard / canary | 同时免爆破 exit_funcs + 破栈保护 |

**一句话**：2.42+ 的通用公式 = `任意分配(tcache_metadata_hijack) → 落 _dl_rtld_lock_recursive / IO 伪造 → exit`。

### 1.3 实战趋势锚点（ptr-yudai 2025 年度精选）【已验证趋势】
- **fastbin 在退场**：snakeCTF 2025 "old school" 是 tcache 被禁用 + fastbin double free，被评价为"再见 fastbin"。2.43 补 fastbin_dup 后此类题更少。
- **内核 data-oriented 成主流**：2025 年内核题"一半以上是 data-oriented（pipe_buffer / 页表）"，不再依赖控制流劫持——`modprobe_path` / Dirty PageTable / PTE overlap 仍是最稳落点。

## 二、ARM64 (AArch64) Pwn 差异速查

> 原报告未覆盖 ARM64 pwn。注意：`arm64-reverse-methodology`（逆向）与本题无关，本节讲的是**漏洞利用**。

### 2.1 调用约定（AAPCS64）
| 角色 | 寄存器 | 说明 |
|------|--------|------|
| 参数 | x0-x7 | 前 8 个整型/指针参数；多余参数入栈 |
| 返回值 | x0 | |
| 帧指针 FP | x29 | |
| 链接寄存器 LR | x30 | 存返回地址（**不是放栈上！**）|
| 栈指针 | sp（=x31） | x31 双角色：sp / xzr |
| 调用者保存 | x0-x18（除 x29/x30） | |
| 被调用者保存 | x19-x28 | 函数内可自由用 |
| 平台/TLS | x18（部分 OS） | |

### 2.2 栈帧与返回机制（与 x86 最大差异）
- 序言：`stp x29, x30, [sp, #-N]!`（**把 LR 也存进栈**，开 N 字节空间）
- 尾言：`ldp x29, x30, [sp], #N`（从栈恢复 LR）→ `ret`
- **`ret` = 跳到 x30**，没有 x86 那种"pop 返回地址"。所以 **ROP 的本质是控制 x30**，而 x30 是被 `ldp` 从栈上加载的。

### 2.3 ARM64 ROP 特征
- **gadget 形态**：`... ; ldp x29, x30, [sp], #0xN ; ret` —— 它从 `sp` 偏移处加载下一个 x30 并返回。每个 gadget 在栈上消耗一个槽位（被它加载的 x30 值），sp 随之推进。
- 溢出布局：`[padding 到 saved x29/x30][gadget1 地址(成为新 x30)][...][gadget2 地址]...`
- **gadget 密度低**：不像 x86 "ret(0xc3)"字节到处都是。用 `ROPgadget --binary x --arch arm64` 或 `ropper --file x --arch AARCH64`，搜 `ldp x29, x30, [sp], #` 模式。
- **ret2csu（通用 ROP）**：libc 的 `__libc_csu_init` 风格函数常含：
  ```
  ldp x19,x20,[sp,#0x10]
  ldp x21,x22,[sp,#0x20]
  ldp x23,x24,[sp,#0x30]
  ldp x29,x30,[sp],#0x40
  ret
  ```
  从栈控 x19-x24 + x30；再配一个把寄存器搬进 x0 的 gadget（或 `blr x3` 类）即可控参数。
- **one_gadget 不支持 arm64**：改用 ROP 调 `system("/bin/sh")` 或 ORW。
- pwntools：`context.update(arch='aarch64')`；`shellcraft` 支持 aarch64。

### 2.4 PAC / BTI / MTE 对 ROP 的影响
| 缓解 | 机制 | 对利用的影响 | 绕过思路 |
|------|------|------------|---------|
| **PAC** (ARMv8.3) | 序言 `paciasp` 给 x30 签名，尾言 `autiasp` 验签 | **栈上 x30 被签名**，直接覆盖任意值→验证失败崩 | ① 重用已签名指针（泄漏合法签好的 LR/函数指针直接用）② 爆破（~16 位，但触发限流）③ 找无 `autiasp` 的 `ret` 路径 ④ **最稳：data-oriented 攻击（改 modprobe_path/数据指针，不碰控制流）** |
| **BTI** (ARMv8.5) | 间接跳转目标须是 `bti c/j/jc`，否则 SIGILL | gadget 入口必须是 `bti` 指令，缩小 gadget 池 | 不致命；PAC+BTI 组合才真正收紧。`ret` 目标 BTI 不查（PAC 管），纯 ret 链在无 PAC 时仍可用 |
| **MTE** (ARMv8.5) | 每 16 字节一个 4-bit 染色 tag，堆随机着色 | OOB/UAF 命中正确 tag 概率仅 1/16→SIGSEGV | 概率爆破（每试 1/16）或泄漏 tag 位 |

> 实操结论：ARM64 现代 pwn **优先走 data-oriented**（覆盖数据/函数指针以外的落点），其次才是"重用已签名指针 + ret 链"。纯栈溢出打 ROP 在 PAC 开启后基本走不通。

## 三、seccomp 沙箱逃逸速查（含代码模板）

> 原报告只提了一句 ORW。本节给完整可跑模板。

### 3.1 分析规则
```bash
seccomp-tools dump ./binary          # 本地静态/运行时 dump
# 远程：把二进制拉下来本地 dump，或抓 prctl/seccomp 调用
```
读输出关注**被禁的 syscall 编号**（x86-64）：
`execve=59`、`open=2`、`openat=257`、`read=0`、`write=1`、`sendfile=40`、`mmap=9`、`mprotect=10`、`memfd_create=319`、`execveat=322`。
禁 execve→走 ORW；连 open 也禁→见 3.4。

### 3.2 ORW 链（shellcode 版，有执行权限时）— x86-64
```python
from pwn import *
context.update(arch='amd64', os='linux')

sc  = shellcraft.pushstr('/flag')
sc += shellcraft.open('rsp', 0)             # open(path, O_RDONLY) -> fd in rax
sc += shellcraft.read('rax', 'rsp', 0x100)  # read(fd, rsp, 0x100)
sc += shellcraft.write(1, 'rsp', 0x100)     # write(1, buf, 0x100)
payload = asm(sc)
```

### 3.3 ORW 链（ROP 版，NX / 无执行权限）— 用 libc gadget
```python
from pwn import *
context.update(arch='amd64', os='linux')
rop = ROP(libc)
flag_str = next(libc.search(b'/flag\x00'))
bss_buf  = libc.bss(0x200)

rop.open(flag_str, 0)           # fd 通常返回 3（继 0/1/2 之后）
rop.read(3, bss_buf, 0x100)
rop.write(1, bss_buf, 0x100)
# 若栈上需要控制 rax 做 syscall，pwntools ROP 会自动找；手动则:
#   rop.raw(rop.find_gadget(['pop rdi','ret'])[0]); rop.raw(flag_str); ...
p.sendline(rop.chain())
```
> 注意：open 返回的 fd 不一定是 3（若进程已开别的 fd）。稳妥法：把 open 返回值存起来再 read，shellcode 版用 `read('rax', ...)` 已自动处理；ROP 版多数情况是 3，不确定时加 `dup2` 或直接假设 3 试。

### 3.4 禁 `open` 时：openat（syscall 257）
```python
AT_FDCWD = -100 & 0xffffffffffffffff   # 用当前工作目录解析相对路径
# shellcraft
sc  = shellcraft.pushstr('/flag')
sc += shellcraft.openat(-100, 'rsp', 0)   # openat(AT_FDCWD, path, O_RDONLY)
sc += shellcraft.read('rax', 'rsp', 0x100)
sc += shellcraft.write(1, 'rsp', 0x100)
# ROP 版
rop.openat(AT_FDCWD, flag_str, 0)
rop.read(3, bss_buf, 0x100)
rop.write(1, bss_buf, 0x100)
```

### 3.5 禁 `open` + `openat`（严格沙箱）的退路
1. **memfd_create（319）**：仅在需要"不落地运行另一个二进制"时用（配 execveat），读 flag 用不上（它创建的是匿名内存文件）。
2. **flag 已被映射**：读 `/proc/self/maps` 找 flag 所在映射 → 直接读那块内存。需 open 能开 /proc（很多沙箱不禁 /proc 的 open）。
3. **sendfile(1, fd, NULL, count)**（40）：能把一个 fd 内容直接发到另一个 fd，省去 read 缓冲。但仍需先有个 fd。
4. **布尔侧信道（最后手段）**：逐字节比较 flag，靠"崩/不崩"或"输出长度差异"做 oracle，一次 1 bit/byte，极慢但万能。

> 决策：`open` 禁→上 `openat`；`openat` 也禁→看能否开 `/proc`；都不行→侧信道。

## 四、实战卡点速查表（具体方法）

| 卡点 | 具体突破方法 |
|------|------------|
| **远程不通、本地通（版本不对齐）** | ① 泄漏一个 libc 地址 → LibcSearcher / libc.rip / libc.blukat.me 反查版本 ② `glibc-all-in-one`：`./update_list && ./download <版本>` 下精确 libc+ld（带符号）③ `patchelf --set-interpreter ./ld-2.xx.so --set-rpath . ./bin` ④ `ldd ./bin` 确认指向本地 ⑤ 本地跑通再打远程。**最常见坑：同 "2.31" 的不同 ubuntu 编译偏移不同**，必须精确到小版本 |
| **one_gadget 全失败** | 约束（如 `[rsp+0x70]==NULL`）不满足。替代：① `system("/bin/sh")`：`rdi = next(libc.search(b'/bin/sh'))` 后 ret 到 system ② 找 `add rsp, 0xN; ret` / `ret` 滑板抬高栈满足约束 ③ 手动 ROP 调 `execve`：rax=59, rdi="/bin/sh", rsi=0, rdx=0, syscall ④ 沙箱题直接换 ORW 链 |
| **格式化字符串任意写偏移** | ① 定位偏移：发 `AAAAAAAA %6$p %7$p...` 数到标记出现，或 `FmtStr` 自动探测 ② **x86-64 规律**：`printf(fmt)` 无额外参数时，printf 读的"参数"是 `rsi(%1) rdx(%2) rcx(%3) r8(%4) r9(%5) 栈(%6起)`；你的 payload 串通常在 `%6` 之后某处 ③ 一键构造：`fmtstr_payload(offset, {addr: value}, write_size='byte')`，自动按 `%hhn` 逐字节写 ④ ARM64 差异：参数先走 `x0-x7`，`%1`~`%8` 读寄存器，`%9` 起才读栈 |
| **栈迁移（stack pivot）** | 目标：把 sp 搬到你放好 ROP 链的可控缓冲（bss/heap）。**x86 方法 A（leave;ret）**：`leave = mov rsp,rbp; pop rbp`。溢出覆写 saved rbp = `chain_addr-8`、saved ret = `leave;ret` gadget → 函数自身尾言 `leave;ret` 后 sp=chain_addr 执行你的链。**方法 B（xchg）**：找 `xchg rsp, rax; ret`（rax 常持堆/bss 指针）。**方法 C（pop rsp）**：找 `pop rsp; ret`，把目标地址放栈上弹入 rsp。**ARM64**：找 `mov sp, x29` 或 `ldp x29,x30,[可控]; ...; mov sp,x29; ret`，控 x29 后迁 sp |
| **safe-linking 算不出堆基址** | ① unsorted bin 的 fd/bk **未加** safe-linking，free 一个进 unsorted 读出堆地址 ② `safe_link_double_protect`：对指针二次 PROTECT 实现任意链接，**无需泄漏**（how2heap 已收录）③ 算式：`poisoned = (heap_base >> 12) ^ target`；`heap_base` 通常页对齐，`>>12` 得到 key |
| **无泄漏途径** | ① 改 `_IO_2_1_stdout_` 的 `_flags`/`_IO_write_ptr` 做部分写，刷出 libc 地址 ② 格式化字符串 `%p` ③ **新（2025）**：BlackHatMEA "Stack Impromptu"——BOF 改 fd 后给对端 socket 发 RST 让对侧 `read` 失败，使未初始化栈缓冲泄漏到另一 socket【ptr-yudai 验证，新颖泄漏原语】 |
| **无 free 函数** | House of Tangerine / sysmalloc_int_free（无 free 利用，2.34+ 通用） |
| **大量 free 但无 edit** | house_of_botcake / tcache_stashing_unlink / House of Water |
| **内核 UAF 只能拿同类对象** | cross-cache（CPU0 分配 / CPU1 释放，跨 partial list 排空到 buddy，再用别的 slab 回收） |
| **userfaultfd 被禁** | FUSE / **MADV_DONTNEED + 并发 mprotect（ptr-yudai 首创，最有效）** / io_uring stall / fallocate |
| **利用链崩在 CFI/PAC** | 改用非函数指针落点：`modprobe_path` / Dirty PageTable / 任意文件写（天然免疫 CFI） |
| **（新）core_pattern 攻击面** | KalmarCTF 2025 "decore"：把恶意程序注册进 `/proc/sys/kernel/core_pattern`，构造畸形 ELF 触发崩溃→core dump 处理器解析符号时 OOB 读 flag。**新增沙箱/提权思路**【ptr-yudai 验证】 |
