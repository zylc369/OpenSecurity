# CTF Pwn 方向技术调研报告（2023-2026）

> 调研日期：2026-06-29
> 验证来源：how2heap (shellphish) 源码、ptr-yudai DiceCTF 2026 writeup、ctf-wiki、CTFtime 索引。
> 标注说明：【已验证】=抓取到一手源码/writeup；【待核对】=链接抓取失败或基于已知知识合成，需后续核对。

---

## 一、近三年 Pwn 方向技术趋势概览

1. **glibc 堆利用"后 hook 时代"定型**：`__malloc_hook/__free_hook` 在 2.34 移除后，IO_FILE 攻击（House of Apple/Cat）、`exit_funcs`、`_rtld_global`、TLS 劫持成为 glibc 2.35-2.43 的标准 RCE 落点；how2heap 已覆盖到 glibc_2.43，并新增 House of Water/Tangerine 等无 free 原语技术。
2. **内核 pwn 进入"页表级"对抗**：`modprobe_path`/`msg_msg` 仍常见，但高难度题转向 Dirty PageTable、cross-cache、PTE overlap；`userfaultfd` 普遍被禁用，迫使选手发明 FUSE / `MADV_DONTNEED`+`mprotect` / `io_uring` / `fallocate` 等扩大竞态窗口的新手段。
3. **缓解机制持续加码**：CET/shadow-stack、ARM PAC/BTI/MTE、CFI、freelist 硬编码（safe-linking、SLUB hardening）迫使利用链更长，"任意写 → 一次劫持即 RCE" 的简单路径变少。
4. **新兴攻击面崛起**：io_uring、WASM/Rust 二进制、seccomp 沙箱逃逸成为 2024-2026 高级赛题的新热点。
5. **工具链趋于成熟稳定**：pwndbg 几乎一统天下，glibc-all-in-one + patchelf 成为本地调试标准配比，angr/符号执行用于自动化约束求解。

## 二、关键技术清单（按重要性排序）

### 2.1 堆利用 —— glibc 2.34+ 新劫持点（最高优先级）

#### 技术 A：House of Apple（IO_FILE wide-data vtable 链）
```
技术名称: House of Apple（_IO_wfile_overflow/_IO_wfile_seekoff 链）
出现频率: 高（glibc 2.35-2.39 主流 RCE 落点）
核心原理: glibc 2.34 移除 hooks 后，伪造 _IO_FILE 的 _wide_data 指针，
         利用 wide data 的 vtable（_IO_wfile_jumps）调用链触发任意函数。
         House of Apple 1 用 _IO_wfile_overflow，Apple 2 用 _IO_wfile_seekoff，
         通过 _IO_wfile_overflow→_IO_wdoallocbuf→vtable->__doallocate 实现 RCE。
关键步骤: 1. 泄漏 libc 地址 + 堆地址
         2. 伪造 _IO_FILE 结构，设置 _wide_data 指向可控堆块
         3. 构造 wide data 的 vtable 指向 _IO_wfile_jumps 偏移，使 __doallocate 落在 system/one_gadget
         4. 触发 FSOP（exit 或 _IO_flush_all_lockp）执行链
关键工具/payload: pwndbg 的 `io` 命令查看 _IO_FILE；伪造模板可复用 roderickchan PoC
适用场景: glibc 2.35+，有任意写 + FSOP 触发条件（exit/printf 触发）
参考来源: roderickchan 博客 2022【待核对】，多次出现于 2023 各赛事（如 SECCON/TCTF）
```

#### 技术 B：House of Cat（_IO_wfile_overflow → malloc 断言）
```
技术名称: House of Cat
出现频率: 中（2023-2024 出现，glibc 2.35-2.37）
核心原理: House of Apple 的演进，利用 _IO_wfile_overflow 内部调用 malloc 的路径，
         通过控制传入 malloc 的 size 触发 malloc 断言或在已知条件下劫持执行。
关键步骤: 1. 伪造 _IO_FILE + wide_data 链
         2. 精心设置触发 _IO_wfile_overflow 的字段
         3. 利用链内部 malloc 调用 + 已有堆布局达成 RCE
关键工具/payload: 同 House of Apple，依赖 pwndbg 调试 IO 结构
适用场景: 无法直接控制 vtable 但能控制 wide_data 字段的场景
参考来源: roderickchan 博客 2023【待核对】
```

#### 技术 C：exit_funcs 劫持
```
技术名称: 劫持 __exit_funcs / initial（atexit 链）
出现频率: 中高（glibc 2.34+ 替代 hook 的常用落点）
核心原理: exit() 会遍历 __exit_funcs 链表调用已注册的 atexit handler。
         每个 entry 的 fn 指针受 PTR_DEMANGLE（旋转异或）保护，
         但只要能泄漏 pointer guard（位于 TLS/ld 偏移）即可伪造。
关键步骤: 1. 任意写覆盖 __exit_funcs->next 或 initial 的 fn
         2. 泄漏/爆破 pointer guard（TLS 中，glibc 2.34+ 偏移固定）
         3. 用 PTR_MANGLE 规则（rol 0x11 后异或）计算目标值写入
         4. 触发 exit / return main
关键工具/payload: pointer guard 计算脚本：`(ptr^key) rol 17` 或用 pwntools rol
适用场景: 有任意写 + 能触发 exit 的 glibc 2.34+ 题
参考来源: 多篇 2023 writeup【已验证机制】，CTFtime pwn writeups 2023-2024
```

#### 技术 D：_rtld_global / _dl_fini 劫持
```
技术名称: 劫持 _rtld_global._dl_rtld_lock_recursive（_dl_fini 链）
出现频率: 中（exit 触发的强力落点）
核心原理: exit→__run_exit_handlers→_dl_fini，_dl_fini 会调用
         _rtld_global._dl_rtld_lock_recursive 函数指针；该指针无加密保护，
         一次任意写即可劫持。也可伪造 link_map 链。
关键步骤: 1. 泄漏 ld/libc 基址（_rtld_global 在 ld.so 数据段）
         2. 任意写覆盖 _dl_rtld_lock_recursive 为 one_gadget/system
         3. 触发 exit
关键工具/payload: pwndbg 查看 _rtld_global 结构
适用场景: 有任意写 + exit 触发，且 _dl_rtld_lock_recursive 可写
参考来源: how2heap 用户态利用讨论 + 2023 多赛事 writeup【机制已验证】
```

#### 技术 E：TLS（TCB/TLS DTV）攻击
```
技术名称: TLS 劫持（覆盖 TLS 中的 pointer guard / canary）
出现频率: 中（2023-2024 上升，glibc 2.34+）
核心原理: TLS 区段（fs 寄存器指向的 tcbhead_t）存放 pointer guard、stack canary、
        dtv 等关键值。堆块若紧邻 TLS（或大堆溢出/越界写到 TLS），
         可同时破坏 canary 绕过栈保护、改写 pointer guard 使 exit_funcs 劫持免爆破。
关键步骤: 1. 确定 TLS 与堆的相对位置（pwndbg: p/x $fs_base）
         2. 通过越界写/UAF 改写 TLS 中的 pointer guard 与目标 fn
         3. 触发 exit 执行伪造 handler
关键工具/payload: pwndbg 的 `tls` / `canary` 命令
适用场景: 堆地址接近 TLS、或有相对写原语的场景
参考来源: 2023-2024 CTF writeup【机制已验证】，ptr-yudai blog
```

### 2.2 House of 系列（近年新增）【已验证 - how2heap 源码】

#### House of Water（glibc 2.36+）【已验证】
```
技术名称: House of Water（smallbin 变体无需爆破）
出现频率: 中（Potluck CTF 2023 Tamagoyaki 首发）
核心原理: 把 UAF 转换为 tcache_perthread_struct 元数据控制（任意地址分配）。
         在紧邻 tcache metadata 处放置 'relative_chunk'（共享 ASLR 第二个 nibble=2），
         用 house_of_botcake/任意 free 制造 0x320、0x330 tcache 条目伪造 FWD/BCK，
         三个块进 unsorted 后大分配归入同一 small bin；
         UAF 改 small_start 的 fd 和 small_end 的 bk 低位指向 fake tcache chunk，
         排空 tcache 后从 small bin 取出，第三次分配返回 fake chunk → 控制 tcache 元数据。
关键步骤: 1. 布置 relative_chunk + small_start + small_end（0x88）+ 填满 0x90 tcache
         2. 伪造 0x331/0x321 头部使 small_start/end 头部进 0x330/0x320 tcache
         3. 三块进 unsorted，大分配归入 small bin
         4. UAF 改 fd/bk 低位为 0x00 指向 tcache metadata
         5. 排空 tcache，分配取回 fake chunk
关键工具/payload: how2heap/glibc_2.36/house_of_water.c
适用场景: 仅 UAF/双 free，无任意写，glibc 2.32-2.39
参考来源: Potluck CTF 2023 Tamagoyaki, 作者 UDP_ctf, smallbin 变体 by 4f3rg4n（how2heap 已收录）【已验证】
```

#### House of Tangerine（glibc 2.34-2.39+）【已验证】
```
技术名称: House of Tangerine（无需 free 的 House of Orange 复活版）
出现频率: 中高（Pico CTF 2024 "High Frequency Troubles" 启发）
核心原理: House of Orange 的现代化版本，不需要调用 free()。
         利用 sysmalloc 中对 top chunk（wilderness）的 _int_free 调用
         （glibc 2.39 malloc.c:2913）：缩小 top size 防止合并、保持 PAGE_MASK 位绕过检查，
         malloc 一个大于可用 top 的块使旧 top 被 _int_free 进 tcache，
         再用 tcache poisoning（需堆泄漏绕过 safe-linking）返回任意地址。
关键步骤: 1. malloc 探测当前 top size
         2. BOF/OOB 改 top size 为页对齐（保留 PAGE_MASK 位）
         3. malloc 较大块触发 sysmalloc，旧 top 经 _int_free 进 tcache
         4. 用堆泄漏计算 safe-linking，改 next 指向目标
         5. 两次 malloc(SIZE_1) 取回目标地址
关键工具/payload: how2heap/glibc_2.39/house_of_tangerine.c；需 5 次 malloc + 3 次 OOB
适用场景: 无 free、有 BOF/OOB 的 glibc 2.34+ 题
参考来源: Pico CTF 2024 High Frequency Troubles（how2heap 已收录）【已验证】
```

#### large_bin_attack（glibc 2.30+，注意 2.42 被补）【已验证】
```
技术名称: large_bin_attack（2.30+ 残留路径）
出现频率: 高（仍是构造任意写"大值"的核心原语）
核心原理: glibc 2.30 加了两处检查，但"新插入块比最小还小"的路径仍不检查 bk_nextsize。
         修改已入 large bin 块的 bk_nextsize = target-0x20，
         插入更小块时执行 victim->bk_nextsize->fd_nextsize = victim，
         即把一个堆地址写入 target。常配合 House of Apple 改 _IO_list_all / wide_data。
关键步骤: 1. malloc 两个同 large bin 但不同 size 的块（p1 大、p2 小）
         2. free(p1)，大分配使其入 large bin
         3. 改 p1->bk_nextsize = &target - 0x20
         4. free(p2)，再大分配把 p2 插入，target 被写为 p2 地址
关键工具/payload: how2heap/glibc_2.35/large_bin_attack.c
注意: **glibc 2.42 已修补**（large_bin_attack < 2.42），2.42+ 需换路【已验证 - how2heap patch 链接】
适用场景: 构造"堆地址写到任意地址"原语，配合 FSOP/IO 攻击
参考来源: 0CTF 2018 heapstorm2，how2heap 收录【已验证】
```

### 2.3 safe-linking 绕过【已验证】
```
技术名称: safe-linking 绕过（PROTECT_PTR/REVEAL_PTR）
出现频率: 高（所有 glibc >= 2.32 tcache/fastbin 利用必经）
核心原理: 2.32 起 tcache/fastbin 的 fd 经 (addr >> 12) ^ ptr 异或保护。
         绕过三法：① 泄漏堆地址算出 key，再异或目标；
                   ② safe_link_double_protect：对指针二次保护实现任意链接（无需泄漏）；
                   ③ 利用 unsorted/small bin 的 fd/bk（未加 safe-linking）泄露堆地址。
关键步骤: 1. 获取堆基址（>>12 部分通常为 0）
         2. poisoned = (heap_addr >> 12) ^ target
         3. 写入 poisoned 到 tcache next
关键工具/payload: pwntools 无内置，自写 encode/decode
适用场景: 所有 glibc 2.32+ 的 tcache poisoning
参考来源: how2heap decrypt_safe_linking.c / safe_link_double_protect.c【已验证】
```

### 2.4 内核利用（kernel pwn）

#### 结构体泄漏原语（KASLR/地址泄露）【机制已验证】
```
技术名称: 常用结构体喷射泄漏（tty_struct / seq_operations / msg_msg / shm_file_data / pipe_buffer）
出现频率: 高
核心原理: UAF 块被这些含函数指针/已知偏移的内核结构体重用，
         读出其内容即可泄漏内核 .text 基址（绕 KASLR）。
         - tty_struct：open("/dev/ptmx")，ops 含 kernel 指针
         - seq_operations：open("/proc/self/stat")，4 个函数指针
         - msg_msg： msgsnd，配合 MSG_COPY 可部分读
         - pipe_buffer：含 anon_pipe_buf_ops，Dirty Pipe/Cow 系列核心
         - shm_file_data：shmat，含多个内核指针
关键步骤: 1. UAF 释放目标 slab
         2. 喷射对应结构体占位
         3. 读 UAF 块泄漏指针，计算 kernel base
关键工具/payload: 各结构体 size 对应 kmalloc-{64,96,...,1024,2k,4k}
适用场景: 任意内核 UAF + 无直接 KASLR 泄漏
参考来源: arttnba3 kernel pwn 系列、ctf-wiki kernel【机制已验证，部分链接待核对】
```

#### msg_msg 任意读/写【机制已验证】
```
技术名称: msg_msg 跨 slab 任意读写
出现频率: 高（仍是内核堆利用主力）
核心原理: msg_msg 头部有 m_ts（消息大小）和 next（指向 msg_msgseg 数据段）。
         UAF 改大 m_ts 实现越界读；伪造 next 指针实现任意地址读；
         配合 pipe_buffer / sk_buff 实现任意写。
关键步骤: 1. msgsnd 申请 msg_msg（可控制 size 选 kmalloc slab）
         2. UAF 改 m_ts 越界读邻居 slab 泄漏
         3. 伪造 next 做任意读，配合其他原语任意写
关键工具/payload: msgsnd/msgrcv，MSG_COPY（无破坏读）
适用场景: 内核 UAF，需跨 slab 读写
参考来源: CVE-2021-22555 等公开利用，ctf-wiki【已验证】
```

#### modprobe_path / poweroff_cmd（经典落点）【已验证】
```
技术名称: 覆写 modprobe_path
出现频率: 高（最简单 LPE 落点）
核心原理: 内核执行未知格式可执行文件时调用 modprobe_path 指定路径的程序（root）。
         任意写把 modprobe_path 改为 "/tmp/x"，触发即可 root 执行脚本。
关键步骤: 1. 泄漏 kernel base 算 modprobe_path 地址
         2. 任意写覆盖 modprobe_path 为脚本路径
         3. 执行畸形格式文件触发，脚本以 root 跑
关键工具/payload: echo -ne "\xff\xff\xff\xff" > /tmp/bad; chmod +x; 运行触发
适用场景: 有内核任意写，modprobe_path 未受保护（CONFIG 非只读）
参考来源: 经典，大量 writeup【已验证】
```

#### Dirty Pipe / Dirty Cred / Dirty PageTable【已验证 + ptr-yudai 验证变体】
```
技术名称: Dirty PageTable（PTE 劫持）及变体
出现频率: 高（2022-2026 高级内核题主流）
核心原理: 页级 UAF 转为页表（PTE）corruption：
         - Dirty PageTable：UAF 一个 page，用 PTE 页回收，改 PTE 物理页号指向任意物理页
         - ptr-yudai DiceCTF 2026 变体：重叠"匿名页 PTE"与"文件页 PTE"，
           使只读文件映射经匿名侧可写 → 任意文件写（改 /etc/passwd 或 /bin/umount）
关键步骤: 1. 制造 page-level UAF（cross-cache 回收到 PCP）
         2. 喷射 PTE 页占位
         3. 改写 PTE 项实现任意物理地址读写 / 任意文件写
         4. 改 creds 或覆盖 suid 二进制
关键工具/payload: mmap 布局 PTI_TO_VIRT，匿名+文件双映射
适用场景: 页级 UAF，cross-cache 可达，有任意文件写落点
参考来源: DiceCTF 2026 cornelslop (ptr-yudai)【已验证】；Dirty PageTable 原论文【待核对】
```

#### userfaultfd 禁用后的竞态窗口扩大【已验证 - ptr-yudai】
```
技术名称: 替代 userfaultfd 的 race-window 扩大法
出现频率: 高（现代内核题 userfaultfd/FUSE 常被禁）
核心原理:
  - FUSE：用户态 FUSE 文件系统可阻塞内核 read，stall copy_from_user
  - SIGALRM/多线程：io 操作中插入可中断点
  - io_uring：异步 IO stall
  - **NEW: MADV_DONTNEED + 并发 mprotect（ptr-yudai DiceCTF 2026 首创）**：
    MADV_DONTNEED 清掉 PTE 使 hash walk 每页缺页，
    同时另一线程疯狂 mprotect 翻转权限，争抢 mmap_lock 使缺页重试，
    可把毫秒级窗口拖到数十秒。两者单独都不够，组合才强。
关键步骤: 1. 大块 mmap 注册，篡改一字节
         2. 线程 A: mprotect 循环翻转 PROT_READ/PROT_READ|PROT_WRITE + usleep
         3. madvise(MADV_DONTNEED)
         4. 线程 B: 触发长遍历（如 check_entry 哈希）使其缓慢
         5. 主线程在窗口内竞态触发漏洞
关键工具/payload: pthread + sched_setaffinity 绑核
适用场景: userfaultfd/FUSE 不可用，需要把 copy_from_user/长遍历拖住的竞态
参考来源: DiceCTF 2026 cornelslop (ptr-yudai)；fallocate 法见 faith2dxy 博客【待核对】
```

#### cross-cache（跨 slab 缓存）攻击【已验证 - ptr-yudai】
```
技术名称: cross-cache / 跨 CPU partial list 排空法
出现频率: 中高（独立 SLUB 缓存/SLAB_NO_MERGE 题必用）
核心原理: 独立缓存（如 SLAB_NO_MERGE）UAF 只能拿到同类对象，需把页归还 buddy 再用别的 slab 回收。
         ptr-yudai 法：CPU0 分配、CPU1 释放，使 CPU0 不断建新 slab 页、CPU1 partial list 堆积，
         node partial 溢出后 empty slab 触发 discard_slab 回 buddy → 用 msg_msg/pipe/PTE 页回收。
关键步骤: 1. sched_setaffinity 绑定不同 CPU 分配/释放
         2. 持续刷满 CPU partial（cpu_partial=120）→ node partial → discard_slab
         3. 页回 buddy/PCP 后喷射目标结构体页回收
关键工具/payload: 多核 + pthread，计算 objs_per_slab / cpu_partial 阈值
适用场景: 独立/不可合并 slab，需跨类型对象重用
参考来源: DiceCTF 2026 cornelslop (ptr-yudai)【已验证】
```

#### SMEP/SMAP/KASLR/KPTI 绕过【机制已验证】
```
技术名称: 缓解绕过组合
核心原理/要点:
  - KASLR：用结构体函数指针泄漏（见 2.4 上）
  - SMEP（内核不可执行用户页）：走 ROP/栈迁移，或用 pt_regs 中的用户栈做 ROP
  - SMAP（内核不可直接读用户页）：用 copy_from/to_user，或用 msg_msg 等内核内数据
  - KPTI：通过 swapgs_restore 跳板返回用户态，或 modprobe_path 路径无需切态
  - CFI（函数指针检查）：用合法函数指针（如 modprobe_path）规避，或利用 setcontext/栈 pivot
关键工具/payload: pt_regs 喷射（trapframe）做内核 ROP；commit_creds(prepare_kernel_cred(0))
参考来源: ctf-wiki kernel、arttnba3【机制已验证】
```

### 2.5 新型 mitigations 绕过【机制已验证，部分待核对】

```
技术名称: Intel CET / Shadow Stack、ARM PAC/BTI/MTE、Kernel CFI
出现频率: 中（2024-2026 新硬件/新内核题）
核心原理:
  - CET shadow stack：硬件维护影子栈校验返回地址，纯 ret 劫持失效。
    绕过：改函数指针而非返回地址（用合法 ptr/stack pivot 到合法帧），或利用"间接分支跟踪"未覆盖的 jmp/call 间接跳转漏洞。
  - ARM PAC：指针带认证签名，伪造指针需泄漏/爆破 PAC。
    绞杀：用签名重用（签好的指针可直接用）、或 PAC bypass（如利用内核已签指针、指针重置点）。
  - ARM MTE：内存染色标签，越界/UAF 命中率降低但可被概率爆破或利用 tag 旁路。
  - Kernel CFI（kCFI/RFP）：校验间接调用目标类型签名。
    绕过：找同签名合法目标（如 commit_creds 与同类函数），或用 set_current_cred 等替代；Dirty Pipe/PageTable 类"不靠函数指针"的攻击天然免疫。
适用场景: 新硬件/新内核题，需调整利用链落点选择
参考来源: 硬件安全研究、Linux kernel hardening docs【机制已验证，具体 PoC 待核对】
```

### 2.6 新兴攻击面【部分已验证，部分待核对】

```
技术名称: io_uring / WASM pwn / Rust pwn / seccomp 沙箱逃逸
出现频率: 中（io_uring、seccomp 高；WASM/Rust 中低，上升中）
核心要点:
  - io_uring：内核异步 IO，子系统大、攻击面广，是 2023-2026 内核 LPE 新热点（CVE 频出）；
    也是 userfaultfd 替代手段（io_uring stall）。
  - seccomp 沙箱逃逸：题目常禁 execve，需用 ORW(open/read/write) 布尔链读 flag；
    高阶题需利用 seccomp 过滤规则漏洞或内核侧绕过。
  - WASM pwn：wasm runtime（V8/wasmtime）堆/UAF、OOB 访问线性内存、类型混淆。
  - Rust pwn：unsafe 块的 UAF/OOB，或逻辑漏洞（如 Vec 越界、Rc/Arc 引用计数）；
    有 LLVM 的 stack protector，但无 GC，堆行为类似 C。
关键工具/payload: seccomp-tools 分析规则；wasm 调试用 wasm-decompile / wasmtime -g
参考来源: 2023-2024 CTF sandbox 题【部分已验证】；io_uring CVE 公告
```

## 四、解题方法论

### 4.1 现代 Pwn 题标准解题流程
1. **侦察**：`checksec` 看 mitigations（NX/PIE/RELRO/Canary/FORTIFY/CET/ASLR）→ `file`/`pwn checksec` → 确定架构与 glibc 版本（本地 `ldd --version`，远程靠 libc 泄漏）。
2. **逆向**：IDA/Ghidra 梳理逻辑，定位功能（菜单题：add/edit/show/delete），找漏洞点（UAF/OOB/DF/格式化字符串/BOF）。
3. **沙箱检查**：远程跑 `seccomp-tools dump`，确认是否 ORW 沙箱、禁了哪些 syscall。
4. **构造原语**：把漏洞升级为 **任意读 / 任意写 / 泄漏**（堆：tcache poisoning/large_bin_attack；栈：格式化字符串/Rop）。
5. **泄漏**：libc 基址（unsorted bin fd / stdout/_IO_2_1_stdout_ 泄漏）、堆基址（tcache fd safe-linking 解密）、ld/TLS。
6. **落点选择**（按 glibc 版本与 mitigations，见 4.2 表）。
7. **触发执行**：one_gadget / system("/bin/sh") / ORW 链读 flag。
8. **调试迭代**：本地用 glibc-all-in-one 对齐远程 libc 版本，patchelf 改 rpath 跑通后打远程。

### 4.2 mitigations 组合应对策略表
| glibc 版本 | 主要变化 | 推荐落点/原语 |
|-----------|---------|--------------|
| ≤ 2.33 | 有 hooks | `__malloc_hook/__free_hook` + one_gadget（经典）|
| 2.34 | 移除 hooks | House of Orange/IO_FILE、exit_funcs、_rtld_global |
| 2.35-2.39 | IO_FILE 成熟 | **House of Apple/Cat**、large_bin_attack+IO、TLS |
| 2.32+ | safe-linking | 泄漏堆地址或 safe_link_double_protect 绕过 |
| 2.42+ | 补 large_bin/fastbin_reverse | House of Water/Tangerine、tcache_metadata_hijacking、exit_funcs/_rtld_global |

| 缓解 | 应对 |
|------|------|
| Full RELRO | 改 hook→不可写，转 IO_FILE/exit_funcs/栈 ROP |
| PIE | 先泄漏代码基址 |
| Canary | 泄漏/爆破/不破坏（TLS 覆盖法）|
| NX | ROP/ret2libc |
| seccomp(ORW) | open→read→write 布尔链，禁 open 则用 openat/memfd |
| SMEP/SMAP | 内核 ROP + pt_regs；modprobe_path 类无需读用户页 |
| CET/CFI | 用合法函数指针/栈迁移；避免纯 ret 劫持 |
| KASLR | 结构体函数指针泄漏 |

### 4.3 常见"卡住"点及突破方法
| 卡点 | 突破 |
|------|------|
| 无泄漏途径 | stdout 结构（_flags/_write_ptr）部分写刷出 libc；或格式化字符串 `%p`；或大堆溢出泄漏 |
| safe-linking 算不出 | 用 unsorted bin fd 拿堆地址，或 safe_link_double_protect 无泄漏法 |
| 远程打不通、本地通 | glibc 版本/patchelf rpath 未对齐，用 glibc-all-in-one 下远程对应版本 |
| one_gadget 全失败 | 换 system("/bin/sh")，或栈迁移 ROP，或 ORW |
| 大量 free 无 edit | house_of_botcake / tcache_stashing_unlink / House of Water |
| 无 free 函数 | House of Tangerine / sysmalloc_int_free |
| 内核 UAF 只能拿同类对象 | cross-cache（跨 CPU partial 排空）|
| userfaultfd 被禁 | FUSE / MADV_DONTNEED+mprotect / io_uring / fallocate |
| 利用链崩在 CFI/PAC | 改用非函数指针落点（modprobe_path、Dirty PageTable、任意文件写）|

## 五、知识库缺失识别（应沉淀 vs 不沉淀）

### 应沉淀（通用可复用原语）
- **堆落点决策树**：glibc 版本 → 可用劫持点（hook/IO_FILE/exit_funcs/_rtld_global/TLS）。
- **House of Apple/Cat/Water/Tangerine** 的伪造模板与触发条件。
- **safe-linking 编解码 + double_protect 无泄漏绕过**。
- **large_bin_attack / tcache_stashing_unlink / house_of_botcake** 原语模板（含版本边界，注意 2.42/2.43 补丁）。
- **内核结构体泄漏表**（结构体→size→泄漏什么→对应 kmalloc slab）。
- **msg_msg 任意读写** + **Dirty PageTable/PTE overlap** 模板。
- **竞态窗口扩大法清单**（FUSE/MADV_DONTNEED+mprotect/io_uring/fallocate）。
- **cross-cache 跨 CPU 排空法**。
- **mitigations 绕过速查表**（第 4.2 节）。
- **标准解题流程 + 卡点突破表**（第 4 节）。
- **工具链安装脚本 + pwntools 模板片段**。

### 不沉淀（特定题目 trick，仅留指针）
- 某具体赛事题目的特殊整数溢出/逻辑漏洞。
- 仅特定 kernel 模块的私有结构布局。
- 纯靠逆向出来的"magic 值"/协议格式。
- 与具体 glibc 小版本编译细节强绑定的偏移（应沉淀"如何动态算"，而非硬编码偏移）。

> 原则：沉淀"能跨题复用的原语 + 决策方法 + 触发条件"，不沉淀"一次性数值/私有逻辑"。
