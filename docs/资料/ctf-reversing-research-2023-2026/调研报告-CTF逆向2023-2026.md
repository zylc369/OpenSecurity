## 一、近三年逆向方向技术趋势概览

1. **"AI 编排 + headless 工具链"成为新范式。** IDA 9.0 落地 **idalib**（C++/Python API 可在 IDA 进程外以原生速度调用、无 RPC/IPC）✅，Ghidra 12.0 将默认脚本引擎从 Jython 切到 **PyGhidra 3.0**（CPython 原生、可 pip 装依赖）✅。这意味着"LLM 不直接操作 GUI，全部走 headless 脚本"的路线（即本项目的核心决策）在工具侧已得到强力支撑——逆向自动化从"调 idat -A -S"可进一步升级到"嵌入 idalib 库内联调用"。

2. **符号执行/合成 去混淆 走向工程化。** Triton 生态沉淀出 **QSynth（程序合成打 MBA/数据编码/虚拟化）** ✅、**Titan（VMProtect 去虚拟化）**、**TritonDSE（探索器）**；angr 内置 `analyses.deobfuscator`（string_obf_finder / api_obf_finder / hash_lookup_api_deobfuscator）与 DREAM/SAILR/Phoenix 反编译结构化算法 ✅。"自动识别混淆模式 + 合成简化"已是标配思路，而非手工 patch。

3. **microcode / IL 层 反混淆成主流。** D-810（eshard，基于 Rolf Rolles 的 HexRaysDeob）在 **Hex-Rays microcode 层**做反混淆 ✅；Binary Ninja 三层 IL（LLIL/MLIL/HLIL）+ Workflow ✅；Ghidra P-Code + Z3 concolic 仿真器 ✅。逆向者越来越多在"反编译器的中间表示层"写变换规则，而不是改汇编。

4. **新语言/新格式逆向刚需化。** IDA 9.0 原生支持 **WASM 反汇编器 + 文件 loader**、RISC-V 反编译器（含 T-Head/玄铁扩展）、nanoMIPS（联发科基带）；FLIRT 自动覆盖 **Golang 1.10–1.23 与 Rust 1.77–1.81** ✅。Go 逆向有 Mandiant **GoReSym** 做符号/类型/moduledata 恢复 ✅。Rust/WASM/Zig/RISC-V/eBPF 频繁出现在 2024–2025 国际赛。

5. **"先 dump 再分析"仍是壳/VM 保护的第一性原则，但手段在升级。** 从纯静态逆向解壳，转向"定位 OEP→动态 dump（IDA 调试器/Frida）→重分析"，且对 VMProtect/强反调试倾向"双机调试/不完整分析 VMP 段"。这与本项目 `packer-handling.md` 的决策树一致。

6. **加密/校验识别 偏向"常量+S-box+熵"自动化。** FindCrypt/Signsrch/YARA 常量匹配 + 符号执行反向求解自定义校验，仍是主轴；白盒密码开始出现在中高难度题。

7. **移动端：国产加固 + Flutter/RN/KMP 成新热点。** ⚠待核对（2024–2025 writeup 较分散）。

## 二、关键技术清单（按重要性排序）

### 技术 1：Hex-Rays Microcode 层反混淆（D-810 / HexRaysDeob）
```
出现频率: 中（OLLVM/商业混淆题）
核心原理: 在 Hex-Rays 反编译器把 asm 提升到 microcode、再降到 ctree 的中间阶段，注册 mba_rules/指令改写规则，把混淆等价变换掉，让最终伪代码直接干净。比"改汇编/打 patch"更安全、可复用。
关键步骤:
  1. IDA 加载目标，F5 确认伪代码被混淆（长 dispatch、MBA、虚假分支）。
  2. 安装 D-810（拷到 .idapro/plugins，pip install z3-solver），Ctrl-Shift-D 选配置（如 default_instruction_only.json）→ Start。
  3. 重新 F5，对比反混淆前后伪代码；必要时自写 .json 规则匹配该题专属模式。
  4. 规则误改导致 IDA 崩溃前，频繁存库（README 明确警告）。
关键工具/脚本: D-810（gitlab.com/eshard/d810）✅、HexRaysDeob（github.com/RolfRolles/HexRaysDeob）✅、genmc（patois/genmc，调试 microcode）。
适用场景: OLLVM（指令替换/虚假控制流/控制流平坦化）、MBA、部分商业混淆器。不适合无 Hex-Rays 的目标。
参考: eshard D-810 README ✅；Hex-Rays 博客《Hex-Rays microcode API vs obfuscating compiler》✅。
```

### 技术 2：控制流平坦化（CFF）通用还原 —— deflat 类脚本 + 符号执行
```
出现频率: 高（OLLVM -fla 几乎每年都有）
核心原理: 平坦化把真实基本块塞进一个大 switch(dispatch) + 状态变量。还原分两派：(a) 静态——定位 dispatcher、识别真实块/虚假块，patch 块间跳转为直接跳转（deflat.py 思路）；(b) 符号执行——用 angr/Triton 跟踪状态变量取值序列，重建真实 CFG。
关键步骤:
  1. 识别 dispatch 主循环与状态变量（通常是 cmp reg, imm；imm 序列即块 id）。
  2. 静态派：用 deflat.py（基于 unicorn/angr）自动定位 main、dispatch、retainer，NOP 掉分支，重写跳转。
  3. 符号派：angr 跑 CFGEmulated，symbolic 执行状态变量，收集每个真实块的后继状态，重建顺序。
  4. 关闭 angr 内存膨胀：限制 active stash、用 LAZY_SOLVES、hook 库函数。
关键工具: angr（CFG + 符号执行）✅；D-810 的 OLLVM 预设规则 ✅；社区 deflat.py（cq674350529 等，⚠待核对具体仓库）。
适用场景: OLLVM 控制流平坦化、 Tigress 部分 pass、自研 CFF。
参考: DIMVA 2018 通用去虚拟化框架（Triton，状态机同构于平坦化）✅。
```

### 技术 3：VM 混淆（虚拟化保护）通用去虚拟化
```
出现频率: 中（VMProtect/Themida/自研 VM 题）
核心原理: VM 把原始指令编译成自定义字节码，运行时由 dispatcher+handlers 解释执行。通用思路（Salwan et al. DIMVA 2018）：①符号执行探索 VM 解释循环，按污点收集每条"字节码→语义"；②把字节码序列提升回原生指令（synthesis/LLVM-IR）；③重编译得到语义等价、体积接近原始的去虚拟化代码。
关键步骤:
  1. 定位 VM entry、dispatch loop、handler 表（多为 switch/jump table）。
  2. 提取每个 handler 的语义（Triton 符号化每条 handler，或 Unicorn 单步模拟记录副作用）。
  3. 对目标字节码做符号路径探索，合成原生操作（QSynth 的 BFS 枚举合成）。
  4. 验证合成结果与原 VM 语义一致（差分测试）。
关键工具: Triton + QSynth ✅（github.com/quarkslab/qsynthesis）；Titan（github.com/archercreat/titan，VMProtect 专用，⚠待核对可用性）；TritonDSE（quarkslab/tritondse）。
适用场景: VMProtect、Themida VM、Tigress 虚拟化 pass、CTF 自研 VM。
参考: DIMVA 2018《Symbolic Deobfuscation: From Virtualized Code Back to the Original》✅；BAR 2020 QSynth ✅；BlackHat USA 2021 Greybox Synthesis ✅。
```

### 技术 4：Mixed Boolean-Arithmetic（MBA）化简
```
出现频率: 高
核心原理: MBA 用恒等式把简单算术表达成复杂的布尔-算术混合式（如 (x^y)+2(x&y)==x+y）。还原靠 SMT 化简或程序合成：把 MBA 子表达式丢给 z3/boolector 求最简等价表达式，或用 QSynth 枚举合成。
关键步骤:
  1. 在反编译里识别典型 MBA 形态（大量 ^ & | 混 +-）。
  2. D-810 内置 MBA 化简规则直接 F5 简化；或对单个表达式写 z3 脚本化简。
  3. 复杂 MBA 用 QSynth 离线合成（需要 Triton 把指令喂进去）。
关键工具: D-810 ✅；z3-solver 直接 simplify()；Triton SMT simplification passes ✅；QSynth ✅。
适用场景: OLLVM 指令替换、商业混淆器的数据流混淆、白盒密码中的查表混淆。
参考: BAR 2020 QSynth（MBA 为核心评测对象）✅；BlackHat USA 2021 ✅。
```

### 技术 5：Go / Rust 二进制符号与结构恢复
```
出现频率: 高（Go 尤其；Rust 逐年增多）
核心原理: Go 即便 strip 也保留 pclntab + moduledata（运行时自用），可据此恢复函数名、行号、类型、源文件路径。Rust 依赖 demangle + FLIRT + 识别 panic/Option/Result 模式。
关键步骤:
  1. Go：跑 GoReSym -t -d -p -strings，导出 JSON（函数/类型/文件）。
  2. 用 IDAPython 脚本 goresym_rename.py 把符号/标签灌进 IDA ✅。
  3. pclntab 被 UPX/魔改时，GoReSym 可基于 runtime_modulesinit 签名扫描并修复 magic ✅。
  4. Rust：靠 IDA 9.0 内置 Rust FLIRT(1.77–1.81) ✅ + demangle + 识别 core::panicking/Option::unwrap 识别关键逻辑。
关键工具: GoReSym（github.com/mandiant/GoReSym）✅；IDA 9.0 Golang/Rust FLIRT + FLIRT Manager ✅；redress（⚠待核对）；golang_loader_assist（⚠待核对）。
适用场景: Go/Rust 写的 crackme、malware loader、云原生工具逆向。
参考: Mandiant《Golang Internals Symbol Recovery》blog ✅；GoReSym README ✅。
```

### 技术 6：WASM 逆向
```
出现频率: 中（2024–2025 国际赛渐多）
核心原理: WASM 是栈式字节码，逆向需还原控制流与函数签名。IDA 9.0 原生提供 WASM 反汇编器 + 文件 loader ✅；wasm2c/wasm-decompile 可转可读 C；wabt 工具链导出 wasm2wat 文本。
关键步骤:
  1. file/identify 确认为 WASM；用 wasm2wat 转文本看结构，wasm-decompile 转伪 C。
  2. IDA 9.0 直接加载（WASM loader），F5（如该段支持）或看反汇编。
  3. 注意 WASM 的 import 表（宿主函数，如 env.*），还原宿主 API 语义是关键。
  4. 字符串常在自定义段或被编译进线性内存，需定位 data 段。
关键工具: IDA 9.0 WASM 模块 ✅；wabt（wasm2wat/wasm2c）；wasm-decompile；wasm-decompile（⚠具体 writeup 待核对）。
适用场景: 浏览器题、Rust-to-WASM、边缘计算/区块链题。
参考: IDA 9.0 发行说明（WASM disassembler & loader）✅。
```

### 技术 7：Unicorn Engine 批量模拟 + Hook 优化
```
出现频率: 高（校验函数还原、VM 单步、密码算法提取）
核心原理: 把目标函数从二进制里抠出来，用 Unicorn 映射代码段+栈+堆，符号化输入批量求值；对热点（如循环体）用 UC_HOOK_CODE 细粒度 hook 收集语义。
关键步骤:
  1. 从 IDA 导出函数字节码与基地址，Uc 映射页。
  2. 设定入参/调用约定，批量灌入候选输入求输出（正向表）。
  3. 对每个 handler/基本块 hook code，记录读写的寄存器/内存 → 重建语义。
  4. 性能优化：用 mem hooks 仅在边界 hook、用 lazy 映射、避免每指令 hook 全表。
关键工具: unicorn-engine；本项目已有 unicorn-templates.md（binary-analysis/knowledge-base）。
适用场景: 自定义密码/校验函数快速建表、VM handler 语义提取、反混淆差分测试。
参考: 本项目知识库 unicorn-templates.md；Triton examples（github.com/JonathanSalwan/Triton/src/examples/python）✅[结构]。
```

### 技术 8：符号执行求解 flag 校验（angr 最佳实践）
```
出现频率: 高
核心原理: 把"输入→成功分支"建模成约束，让 angr 探索到 'Correct' 分支并求解。关键在"避障探索"：避免路径爆炸。
关键步骤:
  1. angr load → project；project.factory.entry_state(bits/symbolic stdin)。
  2. 用 explore(find=success_addr, avoid=[fail_addr1, fail_addr2...])，把所有失败分支加入 avoid。
  3. 防 explode：LAZY_SOLVES、限制 active stash 大小、hook 掉无关库函数（SimProcedure）、用 bitvector 约束输入长度。
  4. found[].posix.dumps(0) 取出满足的输入。
关键工具: angr ✅；Claripy/z3 ✅。
适用场景: 分支密集的 crackme、迷宫题、约束型 flag 校验。
参考: angr 文档（Core Concepts/Symbolic Execution）✅；CTF 经典范例 ⚠待核对具体 writeup。
```

### 技术 9：自定义密码识别（常量 / S-box / 熵）
```
出现频率: 高
核心原理: 自定义密码几乎都改自标准算法，留下常量指纹（AES S-box 0x63...0x16、SM4 FK/CK、MD5 T 表、CRC 多项式）。用常量扫描+熵分析+轮函数结构比对识别变种。
关键步骤:
  1. IDA FindCrypt 插件 / signsrch / KANAL 扫描已知常量与表。
  2. 找不到时：扫 .rdata 高熵 256 字节表（疑似 S-box）、扫魔数（如 CRC 多项式、SHA 初始 H）。
  3. 反向工程轮函数结构，对照标准算法差异（轮数、key schedule 变形）。
  4. 自定义校验用 angr 反向求或 Unicorn 建表爆破。
关键工具: FindCrypt/Signsrch/KANAL；YARA 自写常量规则；本项目 crypto-validation-patterns.md / ecdlp-solving.md。
适用场景: 改了常量/轮数/模式 的对称密码、自定义哈希、CRC/校验和。
参考: 本项目已有 crypto-validation-patterns.md；具体 writeup ⚠待核对。
```

### 技术 10：反调试 / 完整性校验对抗
```
出现频率: 中
核心原理: ptrace 自附加、rdtsc 计时、/proc/self/status(TracerPid)、CRC 完整性校验。对抗思路：①不修逻辑、直接 patch 掉检测函数返回值；②IDA 调试器内用 conditional breakpoint 改返回；③Frida 在 native 层 hook 绕过。
关键步骤:
  1. 反编译定位 anti-debug 函数（特征：ptrace/raise/getpid+读 status/rdtsc 差值）。
  2. 静态 patch：把检测函数 ret 真/或 NOP 调用；CRC 校验则定位校验目标范围与期望值。
  3. 动态：Frida hook 关键 API 返回假值；或 IDA debugger 断点改寄存器。
  4. 完整性校验：若校验 OEP 后代码段，dump 后重分析可绕过；若运行时校验，先 patch 校验函数。
关键工具: Frida（本项目 frida-hook-templates.md / frida-17x-api.md）；IDA debugger；本项目 dynamic-analysis.md。
适用场景: UPX 变种、VMProtect、CTF 自加反调试。
参考: 本项目 packer-handling.md 阶段 2.5/3.5；具体 writeup ⚠待核对。
```

## 四、解题方法论

### 4.1 现代逆向题标准解题流程（与本体系 analysis-planning 对齐）
```
1. 侦察(initial_analysis)：架构/格式/语言/壳/混淆 → die、file、strings、entropy
2. 场景分类：明文逻辑? 自定义密码? 混淆? 加壳? VM? 符号缺失?
3. 预处理：脱壳(dump at OEP)、符号恢复(GoReSym/FLIRT/Lumina)、反混淆(D-810/deflat/symbolic)
4. 定位关键逻辑：strings+xref 到 "Correct/Flag/输入提示"，定位校验函数
5. 算法还原：读反编译；复杂则 Unicorn 建表 / angr 反向 / 手撸 Z3
6. 求解：约束求解 / 爆破 / 数学攻击；能并行用满 CPU、打印时间与 ETA（本项目进化-20260427 教训）
7. 验证：把求出的输入喂回原程序，确认"Correct"
```

### 4.2 按混淆/保护类型的应对策略
| 类型 | 首选策略 | 回退策略 |
|------|---------|---------|
| 无混淆纯逻辑 | 反编译直读 + 必要时 angr 反向 | — |
| 自定义密码 | FindCrypt/熵扫描 → 识别变种 → 数学/表攻击 | Unicorn 建表爆破 |
| OLLVM 指令替换/MBA | **D-810 F5 直接化简** ✅ | z3/QSynth 离线合成 ✅ |
| OLLVM 控制流平坦化 | **deflat.py 静态 patch** 或 angr 符号重建 | 手动 trace 状态变量 |
| 虚拟机(VM) | **Triton+QSynth 符号去虚拟化** ✅ | Unicorn 逐 handler 提语义 |
| 加壳 | 定位 OEP→**IDA 调试器/Frida dump**（本项目 packer-handling 主线） | 静态逆向解壳(≤30min) |
| 反调试/CRC | patch 检测函数 / Frida hook | IDA 条件断点改返回 |
| Go/Rust | **GoReSym 灌符号 + FLIRT/Lumina** ✅ | pclntab 手解 |
| WASM | **IDA 9.0 原生 + wasm2wat/decompile** ✅ | wabt 文本 |
| 移动加固 | **Frida DEX dump**（本项目 3.5b） | blackdex 类工具 |

### 4.3 常见"卡住"点及突破方法
| 卡点 | 突破 |
|------|------|
| 反编译一团乱码 | 先 D-810 反混淆；不行则降到 disasm + Unicorn 仿真单段 |
| 符号执行路径爆炸 | 收紧 avoid、限 stash、hook 库函数、用 backward slice 只切校验相关切片 |
| 找不到主逻辑 | 从提示字符串/网络/文件 IO 反查 xref；或看 TLS callback/init_array |
| 自定义密码认不出 | 不只扫常量，看轮函数结构 + 块大小 + key schedule；高熵表优先 |
| VMProtect 让 IDA 卡死 | 不要全分析 VMP 段；dump OEP 后只分析原始 .text（本项目已有 kernel-driver 指引） |
| Go strip 后没符号 | GoReSym 基于 runtime_modulesinit 扫描修复 pclntab magic ✅ |
| 求解器跑不出来 | 切换 SMT 后端(z3↔bitwuzla)、加约束剪枝、分块求解、用 C 重写热点(本项目 ecdlp-solving 教训) |

### 5.2 题目级 trick —— 不沉淀（举例，警示）
- 某题"把 flag 异或后藏进 BMP 像素" → 不沉淀（一次性）。

- 某题"反调试藏在 TLS callback 检查 IsDebuggerPresent×N 取平均" → 属 anti-debug **通用**模式，可提炼进 A/技术10，但不带题目细节。
> 判据：沉淀的是"识别特征 + 通用应对"，不是"这题怎么做"。
